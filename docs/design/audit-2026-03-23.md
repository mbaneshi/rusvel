# Codebase Audit — 2026-03-23

## Summary

Full audit of RUSVEL codebase covering Rust backend (27 crates), SvelteKit frontend (12 department routes), documentation accuracy, and architecture compliance.

---

## Critical Issues

### C1. Dynamic Tailwind Classes Broken in Production

**File:** `frontend/src/lib/components/chat/DepartmentPanel.svelte`
**Lines:** ~30 occurrences using `bg-{color}-600/20`, `text-{color}-400`, `border-{color}-500/30`, etc.

**Problem:** Tailwind CSS purges at build time. Template-literal class names like `bg-${color}-600/20` are never in the source as literal strings, so the CSS is not generated. Colors render correctly in dev (JIT) but break in production builds.

**Fix:** Replace with a static class mapping object keyed by color name.

**Status:** FIXED

---

### C2. SSE Stream Parsing Duplicated 3x in api.ts

**File:** `frontend/src/lib/api.ts`
**Functions:** `streamDeptChat` (L195-234), `streamCapability` (L471-509), `streamHelp` (L513-551), `streamChat` (L555-612)

**Problem:** Four functions contain identical SSE parsing logic (fetch → reader → decode → split lines → parse JSON → dispatch callbacks). Any bug fix or protocol change must be applied in 4 places.

**Fix:** Extract shared `parseSSE()` helper.

**Status:** FIXED

---

### C3. 12 Redundant Static Department Routes

**Files:** `frontend/src/routes/{forge,code,content,harvest,gtm,finance,product,growth,distro,legal,support,infra}/+page.svelte`

**Problem:** All 12 files are nearly identical (~44 lines each, differing only in config data). The dynamic route `frontend/src/routes/dept/[id]/+page.svelte` already exists and the layout navigates to `/dept/{id}`. The static routes are dead code.

**Fix:** Delete all 12 static route directories. The `dept/[id]` route handles everything via the department registry API. Enhance the dynamic route to pass `helpDescription` and `helpPrompts` from the registry.

**Status:** FIXED

### H3. Seven Engines Not Instantiated in main.rs

**Engines:** finance, product, growth, distro, legal, support, infra

**Situation:**
- All 7 have real domain code (submodules with managers, ~500 lines each)
- `EngineKind` enum has all 12 variants
- `DepartmentRegistry` defines all 12 departments
- Frontend routes serve all 12
- API handles all 12 via parameterized routes
- BUT: only ForgeEngine is constructed in `main.rs`

**Impact:** The 7 engines exist as crates but their domain logic is never called. Chat works for all departments (routed through the generic agent), but engine-specific domain operations don't execute.

**Decision:** Keep — they represent the target architecture. The generic agent handles chat for all departments regardless of whether a dedicated engine is instantiated. Engine instantiation will be added incrementally as domain features mature.

---

### H4. Job Queue Has No Worker Loop

**Exists:** `JobPort` trait, `JobQueue` adapter, `JobStatus` enum, jobs can be enqueued
**Missing:** No `tokio::spawn` worker loop in `main.rs` to dequeue and process jobs

**Impact:** Background jobs sit in DB indefinitely. Any feature requiring async job execution is non-functional.

**Priority:** Implement when first feature needs background processing.

---

### H5. Approval Workflow Incomplete

**Exists:** `ApprovalStatus` enum in domain, ADR-008 documenting the design
**Missing:** No API endpoints (`GET/POST /api/approvals`), no frontend UI

**Impact:** Content publishing and outreach gates (ADR-008) cannot be enforced.

**Priority:** Implement alongside content/outreach features.

## Medium Priority Issues

### M1. `#[allow(dead_code)]` on All Engine Structs

All 7 domain engines and ForgeEngine suppress dead-code warnings on stored port fields. This is intentional — ports are injected for future use as engine features mature.

### M2. DepartmentPanel.svelte Is Large (~666 lines)

Handles agents, skills, rules, MCP, hooks, workflows, capability engine, directories, and events in a single component. Consider splitting into subcomponents when adding features.

### M3. No Analytics Frontend

`/api/analytics` endpoint returns data (agents, skills, rules, events counts) but no frontend component consumes it. The dashboard page could display this.

## Low Priority / Code Quality

### L1. Build Warnings

Run `cargo build 2>&1 | grep warning` to identify current warnings. The earlier analysis found potential unused imports, but verification showed they were false positives — all imports in `chat.rs` and `runway.rs` are used.

### L2. Accessibility Gaps

- Chat messages use `<div>` instead of semantic `<article>` with ARIA roles
- Toggle states (rules/hooks on/off) communicated by color only
- Many form inputs use `placeholder` as label
- No skip navigation links

### L3. Manual Store Subscriptions

Multiple components use `store.subscribe((v) => (local = v))` instead of Svelte's `$store` auto-subscription. This is a style choice but creates more boilerplate. Note: Svelte 5 runes (`$state`, `$derived`) are already used elsewhere, but stores predate this migration.
