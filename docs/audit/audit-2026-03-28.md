## 6. Security Concerns

### 6.1 CORS Allow-All Origins — CRITICAL

| File | Line | Issue | Fix |
|------|------|-------|-----|
| `rusvel-api/src/lib.rs` | 533-536 | `CorsLayer::new().allow_origin(Any).allow_methods(Any).allow_headers(Any)` | Restrict to `localhost:5173` and deployed frontend domain |

Any website can make authenticated cross-origin requests when bearer token auth is enabled.

### 6.2 SQL Injection in rusvel-db — HIGH

| File | Line | Issue | Fix |
|------|------|-------|-----|
| `rusvel-db/src/store.rs` | 172, 190, 207, 223 | PRAGMA table/index/fk injection via `format!` | Validate with `SchemaIntrospector::validate_table_name()` |
| `rusvel-db/src/store.rs` | 279 | ORDER BY injection: `format!(" ORDER BY {o}")` | Validate against `PRAGMA table_info` columns |
| `rusvel-db/src/store.rs` | 278 | SELECT clause injection: `"SELECT {cols} FROM..."` | Validate column names or allow only `*` |

**Note:** The API layer (`db_routes.rs`) has its own validation via `SchemaIntrospector`, so these are only exploitable if `store.rs` methods are called from other code paths without validation.

### 6.3 Path Traversal in Built-in Tools — MEDIUM

| File | Line | Issue | Fix |
|------|------|-------|-----|
| `rusvel-builtin-tools/src/file_ops.rs` | 39-73 | `read_file` — no path validation | Add `canonicalize()` + allowed-base check |
| `rusvel-builtin-tools/src/file_ops.rs` | 100-121 | `write_file` — creates arbitrary dirs + writes | Same |
| `rusvel-builtin-tools/src/file_ops.rs` | 152-184 | `edit_file` — no path validation | Same |
| `rusvel-builtin-tools/src/file_ops.rs` | 211-243 | `glob` — user-controlled base + pattern | Canonicalize base, validate results in-bounds |
| `rusvel-builtin-tools/src/file_ops.rs` | 274-306 | `grep` — passes unvalidated path to `rg` | Validate path is within allowed directory |

### 6.4 Arbitrary SQL Endpoint — MEDIUM

| File | Line | Issue | Fix |
|------|------|-------|-----|
| `rusvel-api/src/db_routes.rs` | 251-281 | `POST /api/db/sql` accepts arbitrary SQL; only `PRAGMA query_only = ON` guards | Require explicit authorization; document risk |

### 6.5 Passing Checks

- **Secrets/tokens in logs:** PASS — No sensitive values logged
- **Bearer token handling:** PASS — Silent comparison, no token leakage

## 7. Performance Red Flags

### 7.1 Blocking Database I/O in Async Context — CRITICAL

| File | Line | Issue | Fix |
|------|------|-------|-----|
| `rusvel-db/src/store.rs` | 385-484 | Synchronous rusqlite operations inside `async` trait impls block tokio executor | Wrap in `tokio::task::spawn_blocking()` or migrate to `tokio-rusqlite` |

All `EventStore`, `ItemStore`, `SessionStore` async methods acquire a `Mutex<Connection>` and run synchronous I/O. Under load, this starves the async runtime.

### 7.2 Unnecessary Vector Allocation — MEDIUM

| File | Line | Issue | Fix |
|------|------|-------|-----|
| `rusvel-api/src/chat.rs` | 344 | `msgs[msgs.len() - limit..].to_vec()` clones truncated messages | Use `msgs.drain(..msgs.len() - limit)` or return a slice |

### 7.3 Passing Checks

- **`std::fs::` in async code:** PASS — Uses `tokio::fs` where async
- **N+1 queries:** No obvious patterns found in store implementations
